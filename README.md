# kangaroo-wild

Pollard's kangaroo wild search on short interval using a precomputed scored DP database for fast ECDLP key retrieval on the secp256k1 curve.

## Overview

This program solves the Elliptic Curve Discrete Logarithm Problem (ECDLP) on secp256k1 using a two-phase kangaroo approach:

1. **Tame phase** (separate program, not included here): precomputes a scored distinguished point (DP) database.
2. **Wild phase** (this program): launches wild kangaroos that walk the curve and collide with the tame database to recover private keys.

The scored DP technique significantly reduces the expected number of steps compared to a standard random walk, by biasing the database toward high-quality distinguished points.

## Requirements

- Linux (x86_64)
- GCC with support for `__int128` and inline assembly (BMI2/mulx)
- pthreads

## Compilation

```bash
gcc -O3 -march=native -falign-functions=32 -falign-loops=32 kangaroo_wild.c G_stub.c -o kangaroo_wild -pthread -lm -flto
```

## Required database files

The current precomputed databases cover the range **2^74 – 2^75**. Using the extend mode, they can also be used to search in larger ranges (e.g. 2^75 – 2^76) by automatically partitioning the search space.

### Choosing a database size

| DB size | Fingerprints in RAM | Approx. steps to solve | Best for |
|---------|-------------------|----------------------|----------|
| 2^28 | ~1 GB | ~350M | Machines with 4–8 GB RAM |
| 2^29 | ~2 GB | ~250M | Machines with 8–16 GB RAM |
| 2^30 | ~4 GB | ~200M | Machines with 16+ GB RAM |

Larger databases reduce the expected number of steps per key but require more RAM for the fingerprint index (always locked in memory). The tame DB itself can stay on SSD with the `-ssd` option.

Place all 4 database files in the same directory as the executable.

## Usage

```bash
# Solve keys listed in public_keys.txt (one compressed pubkey per line)
./kangaroo_wild

# Solve keys from a custom file
./kangaroo_wild mykeys.txt

# Solve a single key
./kangaroo_wild single 02abc123...

# Solve a key in an extended range (larger than the DB covers)
./kangaroo_wild extend 02abc123... 66 67

# Solve multiple keys simultaneously
./kangaroo_wild parallel public_keys.txt

# Benchmark with random keys
./kangaroo_wild test

# Generate random test keys
./kangaroo_wild gen_test 100

# Show help
./kangaroo_wild help
```

### Storage options

The `-ram` and `-ssd` flags control how the tame database is loaded:

```bash
# SSD mode (default): fingerprints and bucket offsets locked in RAM,
# tame DB stays on SSD and is loaded on demand via mmap.
# Use when RAM is limited or the DB is large.
./kangaroo_wild -ssd

# RAM mode: all data locked in RAM, including the tame DB.
# Fastest, but needs enough RAM for everything.
./kangaroo_wild -ram

# The flag can appear anywhere on the command line:
./kangaroo_wild -ssd single 02abc123...
./kangaroo_wild parallel public_keys.txt -ram
```

| Mode | RAM usage (2^30 DB) | Performance |
|------|-------------------|-------------|
| `-ssd` (default) | ~4 GB (fingerprints + offsets) | ~97% of RAM speed |
| `-ram` | ~12 GB (all data locked) | 100% |

### Tuning parameters

Worker threads and batch size can be set only before the compilation:

```
#define DEFAULT_NUM_WORKERS 16
#define DEFAULT_BATCH_K     80
```

The total parallelism is `C = workers × batch`. Higher C increases throughput but also increases the latency overhead per key.

## Public key format

Input keys must be in **compressed** format (66 hex characters starting with `02` or `03`):

```
02a1633cafcc01ebfb6d78e39f687a1f0995c62fc95f51ead10a02ee0be551b5dc
```

## How it works

The wild kangaroos start from random positions near the target public key and perform a pseudorandom walk on the elliptic curve. When a kangaroo lands on a distinguished point (determined by leading zero bits in the x-coordinate), the program checks for a collision with the precomputed tame database. A collision reveals the discrete logarithm (private key).

Key features:
- **Scored DP database**: the tame phase selects DPs that maximize collision probability, improving efficiency over uniform random DPs.
- **Batch inversion**: Montgomery's trick is used to batch modular inversions, significantly speeding up point additions.
- **Cycle detection**: multiple layers of cycle detection (history buffer, local fingerprint buffer, visited DP tracking) prevent kangaroos from wasting steps in loops.
- **Escape mechanism**: when a cycle is detected, the kangaroo is displaced by a large random jump to escape the cycle.
- **Extend mode**: for key ranges larger than the database covers, the program automatically partitions the search space and searches all partitions.
- **Parallel mode**: multiple independent keys can be searched simultaneously, with workers distributing across all unsolved keys.
- **SSD-friendly**: the tame DB can be left on SSD with minimal performance loss, reducing RAM requirements.
